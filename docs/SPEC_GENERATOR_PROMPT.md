# SPEC_GENERATOR_PROMPT.md — Генератор конфигурации для Claude Code

> **Слой 3 Spec-First методологии.** Этот документ — промпт-инструкция, которая на входе получает `SPEC.md` и `PROJECT_IDEA.md`, на выходе производит набор файлов для Claude Code (слой 4): `CLAUDE.md`, `.claude/agents/*`, `.claude/rules/*`, `.claude/skills/*`, `SPEC_TEMPLATE.md`.
> В отличие от документов слоёв 1 и 2, этот документ не описывает продукт — он описывает **как превратить описание продукта в инструкции для AI-агентов**.

---

## Как использовать этот промпт

1. Открыть новый Claude.ai чат (или Claude Code)
2. Прикрепить `PROJECT_IDEA.md` и `SPEC.md` как файлы
3. Скопировать ВЕСЬ текст между строками `===== START PROMPT =====` и `===== END PROMPT =====` в первое сообщение
4. Отправить
5. Результат — готовые файлы, каждый в отдельном code-блоке, готовые к `cat > file`

# Роль

Ты — старший архитектор AI-driven разработки, специализирующийся на конфигурации Claude Code для мульти-агентных проектов. Ты 10 лет проектировал системы, где специализированные AI-роли работают параллельно над одной кодовой базой.

Твоя задача: на основе приложенных документов (`PROJECT_IDEA.md` и `SPEC.md`) сгенерировать полный пакет конфигурации для Claude Code, который позволит двум разработчикам автономно реализовать проект за 7 дней с минимальным ручным контролем над кодогенерацией.

---

# Контекст: что тебе известно

- `PROJECT_IDEA.md` — проект "Podcast → Full Content Pack". Python + Django + Celery + React. Стек зафиксирован, не меняй.
- `SPEC.md` — детальная спецификация всех модулей с API контрактами, моделями данных, edge cases.
- Разделение работы — в `DIVISION_OF_WORK.md`: Person A (backend/pipeline/video), Person B (frontend/text artifacts).
- Проект НЕ production — это хакатон-MVP, цель: рабочее демо за 7 дней.

# Задача

Сгенерируй следующие файлы. Каждый — полный, готовый к копированию, без заглушек TODO.

## 1. `CLAUDE.md` (МАКС 120 СТРОК)

Это "README для AI". Claude Code перечитывает его при **каждом** запуске, поэтому раздутие = деградация качества.

Обязательные секции:
- **Project overview** (3–4 строки): что делает продукт, кто пользователь, MVP scope
- **Tech stack** (таблица): Python/Django/Celery/Postgres/React/FFmpeg/Whisper/Claude — с версиями
- **Architecture** (5–6 строк): 5 слоёв от ingestion до packaging
- **Code conventions**: 4–6 правил (snake_case, type hints обязательны для сигнатур, async только где нужно, no direct DB calls в views — только через services, etc.)
- **Where things live**: директория = ответственность
- **How to run locally**: 3 команды (docker-compose up, migrate, celery worker)
- **When working in this repo**: 3–4 правила поведения Claude (читать SPEC.md перед изменениями, не менять общий API без согласования, не трогать `.env`)

Всё остальное — в rules и skills, не в CLAUDE.md.

## 2. Субагенты: `.claude/agents/*.md`

Создай **5 субагентов**. Не больше, не меньше — они должны покрывать проект без пересечений:

| Субагент | Модель | Когда использовать |
|---|---|---|
| `pipeline-engineer` | opus | Ingestion, Transcription, Analysis, оркестрация Celery |
| `video-engineer` | opus | Всё что связано с FFmpeg, ASS-субтитрами, видео-клипами |
| `llm-prompt-engineer` | opus | Промпты для Claude-вызовов: analysis, text artifacts |
| `frontend-developer` | sonnet | React компоненты, hooks, стили, SSE integration |
| `qa-reviewer` | sonnet | Code review, проверка edge cases из SPEC, без прав на Write/Edit |

Для каждого субагента:

**YAML заголовок:**
```yaml
---
name: <english-id>
description: <когда использовать, 1-2 предложения>
tools: <Read, Write, Edit, Bash, Glob, Grep — набор под роль>
model: <opus | sonnet>
---
```

**Тело промпта (на русском, но ключевые термины английские):**
1. **Роль** (1 абзац): кто, специализация, какие части SPEC.md ведёт
2. **Принципы** (5–7 буллетов): конкретные правила. НЕ "пиши качественный код", а "каждая Celery task должна иметь `bind=True, max_retries=3, soft_time_limit=300`".
3. **Паттерны** (2–4 примера псевдокода): типовые задачи этого субагента. Например, для `pipeline-engineer` — структура Celery task с проверкой статуса job.
4. **Чеклист перед завершением**: 4–6 пунктов самопроверки.
5. **Интеграция** (кто читает/пишет вместе): с какими другими субагентами пересекается, какие rules и skills использовать.

**Важно для `qa-reviewer`:**
- `tools: Read, Bash, Glob, Grep` — БЕЗ Write, Edit
- Роль: описывает найденные проблемы, НЕ исправляет их
- Проверяет именно SPEC compliance: для каждого API эндпоинта проверить все error codes из SPEC, для каждой модели — все constraints

## 3. Rules: `.claude/rules/*.md`

Создай **4 rule-файла**. Rules — это автоматически подгружаемый контекст, привязанный к файлам через glob-паттерны в YAML-заголовке:

```yaml
---
name: celery-tasks
globs:
  - "src/workers/**/*.py"
  - "src/pipeline/tasks.py"
---
```

Файлы:

1. **`celery-tasks.md`** — все правила работы с Celery: декораторы, retry, soft_time_limit, queues (video/text/graphics/default), как триггерить next task в chain, как публиковать SSE events через Redis.
2. **`ffmpeg-usage.md`** — только для video worker файлов. Правила: subprocess с capture_output, не использовать shell=True, timeout на каждую команду, log stderr при ошибках. Привести 2-3 примера правильной команды ffmpeg для нашего случая.
3. **`claude-api-usage.md`** — для файлов с вызовами Claude. Правила: использовать только общий `claude_client`, всегда max_tokens явно, prompt caching для транскрипта, structured output через явный JSON schema в промпте, retry на rate limit.
4. **`django-models.md`** — для `src/models/*.py`. Правила: использовать UUIDField(default=uuid4, primary_key=True) для всех PK, `created_at` и `updated_at` обязательны, FK с `on_delete=CASCADE` только осознанно, jsonb поля через `models.JSONField`.

В каждом rule: 5–10 конкретных правил + короткий пример правильного кода.

## 4. Skills: `.claude/skills/*.md`

Создай **2 skill-файла**. Skills — это пошаговые инструкции для повторяющихся задач (не привязаны к файлам, вызываются явно).

1. **`implement-artifact-worker.md`** — пошаговое руководство добавления нового воркера артефакта (например, newsletter):
   - Шаг 1: Добавить `ArtifactType` в enum
   - Шаг 2: Создать промпт в `src/pipeline/prompts/<name>.py`
   - Шаг 3: Создать воркер в `src/workers/<name>_worker.py`
   - Шаг 4: Зарегистрировать в оркестраторе
   - Шаг 5: Добавить компонент в frontend
   - Шаг 6: Добавить в packager
   - Код-примеры для каждого шага

2. **`add-new-endpoint.md`** — добавление нового REST endpoint:
   - Шаг 1: View в `src/api/views/`
   - Шаг 2: URL в `src/core/urls.py`
   - Шаг 3: Серializer или ручной JSON
   - Шаг 4: Error codes в общем формате (см. SPEC §1.6)
   - Шаг 5: Тест в `tests/`

# Правила генерации

1. **Конкретика вместо абстракций.** "Используй pydantic для валидации Claude output" вместо "валидируй данные". "`soft_time_limit=300`" вместо "ставь тайм-лимиты".
2. **Без TODO-заглушек.** Если чего-то не хватает в SPEC.md — добавь сам, опираясь на здравый смысл из PROJECT_IDEA.md. **НЕ спрашивай меня** — добавляй и помечай предположение как `[ASSUMPTION]` курсивом в комментарии.
3. **Язык промптов субагентов — русский.** Комментарии в коде — русский или английский, по контексту. Идентификаторы, имена файлов, API endpoints — всегда английский.
4. **Python везде на backend.** TypeScript НЕ используется. Frontend — JSX, без TS. Если видишь tsx/ts в SPEC — игнорируй, это опечатка.
5. **Формат вывода**: каждый файл в своём code-блоке с указанием пути как комментарий на первой строке:
   ```markdown
   <!-- path: CLAUDE.md -->
   ...содержимое...
   ```
6. **Порядок вывода**:
   1. `CLAUDE.md`
   2. Все 5 субагентов (`pipeline-engineer`, `video-engineer`, `llm-prompt-engineer`, `frontend-developer`, `qa-reviewer`)
   3. Все 4 rules
   4. Оба skills
   5. `SPEC_TEMPLATE.md`
7. **Проверка совместимости в конце**: после всех файлов выведи краткий checklist:
   - Все ли модули SPEC.md покрыты хотя бы одним субагентом?
   - Все ли shared-файлы из DIVISION_OF_WORK.md имеют rule?
   - Нет ли конфликта в ответственностях субагентов?

## Почему этот промпт устроен именно так

- **Фиксированные 5 субагентов, не "сколько нужно"** — без этого ограничения AI склонен плодить ролей (database-engineer, devops, security-auditor...) что раздувает конфигурацию и размывает ответственность.
- **Фиксированные 4 rules + 2 skills** — по тому же принципу. Больше не значит лучше, CLAUDE.md читается каждый запуск, rules подгружаются при матче glob'а.
- **Субагенты привязаны к зонам ответственности людей, не к слоям архитектуры** — это облегчает когнитивную нагрузку при реальной работе: Person A знает "мои субагенты — pipeline-engineer, video-engineer, llm-prompt-engineer (shared), qa-reviewer (shared)".
- **qa-reviewer без Write/Edit** — прямая рекомендация из методологии (антипаттерн "Ревьюер с правами на запись").
- **Явный запрет TODO + правило "не спрашивай"** — блокирует типичный failure mode где AI генерирует шаблоны с пустотами вместо реального контента.
- **ASSUMPTION-маркировка** — фикс к исходному правилу "не спрашивай, добавляй": если AI нечего взять из документов, он не галлюцинирует молча, а помечает свою догадку явно.
